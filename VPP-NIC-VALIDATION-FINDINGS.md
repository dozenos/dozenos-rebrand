# `test-vpp` NIC-validation failure — investigation notes (2026-07-20)

**Status: PARKED.** Waiting for an upstream VyOS nightly release to compare
against. Resume from "The one experiment left" at the bottom.

Tracking issue: dozenos-nightly-build#1.

## Symptom

Every VPP smoketest from `test_02_vpp_vxlan` onward errors on `cli_commit()`:

```
dozenos.configsession.ConfigSessionError: [ vpp ]
NIC used by "eth1" is not validated for VPP on DozenOS. Using it is
unsafe and unsupported and will void support for the entire system. To
proceed at your own risk, enable: "set vpp settings allow-unsupported-nics".
```

Upstream's `test_smoketest_cli_vpp` passes 34/34 on the **same** vyos-1x
commit (`fb6f19f`). Not a flake on their side — three consecutive runs green
(2026-07-19 19:49, 2026-07-20 09:03, 09:04).

This only became visible on 2026-07-20. Before that the leg died earlier, in
GRUB console navigation, and never reached the VPP suite at all (fixed in
dozenos-rebrand `ed2c3d9`).

## Mechanism (proven on hardware, reproducible in ~30 s)

When VPP takes a NIC it binds the PCI device to `vfio-pci` and creates a
**host tap with the same interface name**. Live box, mid-failure:

```
eth1 driver: tun          /sys/devices/virtual/net/eth1   <- VPP's tap
pci 00:13.0 -> vfio-pci                                   <- the real NIC
```

`vpp.py`'s `get_config()` re-detects an interface's identity whenever it is
not in the effective config, and at that moment the live `eth1` **is the
tap**, not the NIC. The good persisted values get overwritten with garbage.
Probe inserted into `_is_device_allowed()`:

```
persist_config = {"eth1": {"original_driver": "tun", "dev_id": "tap", "pci_id": null, ...}}
```

`pci_id` is then `None`, so the check falls back to the driver, and
`SUPPORTED_DRIVERS` contains only `hv_netvsc` — virtio survives solely via
`SUPPORTED_PCI_IDS`. Verdict: rejected.

Confirmed the PCI path itself is healthy on the same box, so this is not
"virtio is unsupported":

```
dev_id : 0000:00:12.0
pci_id : 1af4:1000
in SUPPORTED_PCI_IDS: True
```

`_is_device_allowed()` by persist-store shape:

| shape | result |
|---|---|
| populated (fresh add) | True |
| entry wiped | False |
| entry present, no `pci_id` | False |
| wiped + `allow-unsupported-nics` | True |

### What sets it off

`test_01` passes; everything after fails. The difference is that `test_02`
onward **deliberately commit an invalid config first**:

```python
self.cli_set(vxlan_path + [interface_vxlan, 'remote', source_address])
with self.assertRaises(ConfigSessionError):
    self.cli_commit()          # expected to fail
```

`vpp.py`'s `apply()` runs and takes the NIC; a later validation aborts the
commit. **The config rolls back, the dataplane does not.** Every later commit
sees the tap.

A plain add → delete → re-add cycle passes cleanly — the failed commit is
required to reproduce. And it does not self-heal: `eth1` stays `tun` for 30+
seconds, and after deleting all VPP config and committing successfully the
dataplane is still not restored:

```
vpp configured: False     <- config is gone
eth1 driver: tun          <- tap still there
pci 00:13.0 -> vfio-pci   <- NIC still held
vpp service: active       <- still running
```

Only a reboot recovers it.

## Ruled out

Everything below was **compared and found identical** to transformed
upstream (copyright headers excluded):

| checked | files | result |
|---|---|---|
| dozenos-1x VPP code | 57 | identical |
| interface/op-mode definitions (incl. conf_mode priority) | 27 | identical |
| smoketest `test_vpp.py` | 1 | identical |
| VPP patch functional content | 32 | identical (only author email + one added copyright comment) |
| VPP build recipe | 2 | identical |
| kernel config fragments | 27 | identical |

Also identical: VPP package version (`25.10.0-55~…20260409073441`), kernel
version (6.18.38), QEMU args (`-smp 4 -m 8G`, 8× `virtio-net-pci`), kernel
cmdline (`mitigations=off isolcpus=2-3 hugepagesz=2M hugepages=1800`), and
the `check-qemu-install` invocation.

The differing VPP version hash suffix (`eb18ae8f2` vs `7a02d4dae`) is benign:
`git am` re-commits the patches under a different author email, so the tree
matches but the commit sha does not.

**`offload` is NOT the cause.** Our log carries 33 `offload option in ethN
settings is not supported by VPP` warnings and upstream's carries zero, which
looked like a lead. It is auto-generated by
`src/activation-scripts/20-ethernet-offload.py`, which mirrors the kernel's
actual enabled offloads into the CLI config. Deleting `offload` from `eth1`
and re-running the reproduction on a freshly booted box **still fails
identically**.

**Not the CI environment either.** The real upstream `test_vpp.py` fails the
same way on a plain Proxmox VM over SSH — no GitHub runner, no nested
virtualisation, 2 NICs instead of 8, no cross-test residue:

```
Ran 2 tests in 21.042s
FAILED (failures=2, errors=2)
```

Notably `test_01` fails there too, where it passes in CI — so our image trips
this *more* readily than the CI VM does, but the defect is in the image.

## The one experiment left

Boot an **upstream VyOS rolling ISO** and run the same `test_vpp.py`.

- **Also fails** → an upstream bug that some condition in their CI masks;
  file it upstream with the reproduction above.
- **Passes** → the divergence is in our image assembly (package set,
  live-build config, build type — note theirs is `-integration`, ours
  `-rolling`), which is now a small enough space to bisect.

Every comparable *source* input is identical, so the answer is in the built
artefact or how it is assembled, not in code we can diff.

## Interim workaround (not applied)

`set vpp settings allow-unsupported-nics` in the smoketest's `setUp` is
upstream's own escape hatch and was verified to flip
`_is_device_allowed()` to True. It is defensible on its own terms — the
smoketest runs on QEMU virtio NICs, legitimately "unvalidated" hardware — and
would let the leg run to completion. Deliberately **not** applied yet: it
would mask whatever the real divergence is while it is still unknown.

## Reproduction

`test_vpp.py` needs 8 GB RAM, 4 physical cores, `isolate-cpus`, hugepages,
and a second NIC. On a DozenOS box:

```
set system option kernel memory hugepage-size 2M hugepage-count 1800
set system option kernel cpu isolate-cpus 2-3
# save, reboot, then:
cd /usr/libexec/dozenos/tests/smoke/cli
sudo python3 test_vpp.py TestVPP.test_01_vpp_basic TestVPP.test_02_vpp_vxlan
```

Reboot between attempts — a failed run leaves the NIC bound to `vfio-pci`
and poisons every subsequent one.
