# SB-SIGNING.md — Secure Boot MOK signing flow (item #10)

Authoritative spec for how a DozenOS ISO gets a UEFI-Secure-Boot-capable,
MOK-signed kernel. Covers: which mechanism signs what, where the real key
material comes from, the local (no key) vs CI (signed) contrast, and the
verification a maintainer runs post-build. See `CI-SECRETS.md` for the
exact secret names/formats, `data/certificates/README.md` (shipped via
`overlay/new-files/`) for the on-disk contract, and
`release/inject-mok-cert.sh` for the injection helper this document
describes the usage of.

## 1. Two separate signing chains — do not conflate them

DozenOS (inherited from upstream VyOS) has **two independent** module/kernel
signing mechanisms that use **two different keys**. This is standard Debian/
Linux Secure Boot architecture, not a DozenOS-specific design:

| | Kernel module (`.ko`) signing | Kernel EFI (`vmlinuz`) signing |
|---|---|---|
| **Key** | Per-build **ephemeral** RSA key, generated by the kernel build itself (`scripts/package-build/linux-kernel/build-kernel.sh`, `EPHEMERAL_KEY`/`EPHEMERAL_PEM` under `/tmp`, sourced from `CONFIG_MODULE_SIG_KEY`) | The **MOK** (Machine Owner Key) — a real, persistent keypair the user owns, injected from CI secrets |
| **Public half trusted via** | The kernel's own **builtin trusted keyring** (`CONFIG_SYSTEM_TRUSTED_KEYRING=y`) — the cert is baked into the kernel image itself, so the kernel trusts its own embedded key automatically, no enrollment needed | **shim + UEFI `db`/MOK list** — the firmware/shim chain must have the MOK's public cert enrolled (`mokutil`/`MokManager`, a one-time manual step on real hardware) |
| **Signed by** | `scripts/package-build/linux-kernel/sign-modules.sh` (calls `${KERNEL_DIR}/scripts/sign-file`), invoked from **every** OOT module's own build script, right after `make modules_install`, at **package-build time** — long before live-build even starts | `data/live-build-config/hooks/live/93-sb-sign-kernel.chroot` (`sbsign`), at **live-build chroot-hook time**, near the end of the ISO build |
| **Never touches** | `data/certificates/` / `release/inject-mok-cert.sh` at all — this key is generated and discarded per build, never an org secret | N/A (this row exists to reject a common misreading: neither chain needs the *other's* key) |

**What this means for item #10 / `inject-mok-cert.sh`:** it only ever
concerns the **right-hand column** (the MOK, for `vmlinuz` signing). Module
signing is already fully solved at package-build time and needs no CI-secret
wiring — it works identically in local and CI builds.

## 2. Module-signing coverage (left-hand column, for completeness)

Every OOT module build script calls `sign-modules.sh <dir-containing-.ko-files>`,
which itself **globs** (`find ${MODULE_DIR} -type f -name \*.ko`) rather than
listing modules by name — so it automatically covers whatever `.ko` files
that recipe produced, with no per-module allowlist to keep in sync:

| Recipe | Modules | Calls `sign-modules.sh` |
|---|---|---|
| `build-intel-nic.sh` | `i40e`, `iavf`, `ice`, `ixgbe`, `ixgbevf` | yes (line 64) |
| `build-realtek-r8126.py` | `r8126` | yes (line 81) |
| `build-realtek-r8152.py` | `r8152` | yes (line 82) |
| `build-jool.py` | `jool`, `jool_common`, `jool_siit` | yes (line 90) |
| `build-ipt-netflow.sh` | `ipt_NETFLOW` | yes (line 62) |
| `build-nat-rtsp.sh` | `nf_conntrack_rtsp`, `nf_nat_rtsp` | yes (line 33) |
| `build-accel-ppp-ng.sh` | `ipoe`, `vlan_mon` (accel-ppp-ng OOT modules) | yes (line 62) |
| `build-intel-qat.sh`, `build-mellanox-ofed.sh` | QAT / OFED modules | yes |

No gap found against the documented `E: Module not signed` set. Because the
signing calls happen at package-build time (glob over that recipe's own
output dir), a module would only be *missed* if a **new** OOT recipe were
added without also calling `sign-modules.sh` on its `.ko` output directory —
that is a package-build-script review concern, not something
`93-sb-sign-kernel.chroot` or `inject-mok-cert.sh` can catch or fix.

## 3. `93-sb-sign-kernel.chroot` — what it actually does

```sh
KERNEL_KEY="/var/lib/shim-signed/mok/dozenos-dev-2025-linux.key"
KERNEL_CERT="/var/lib/shim-signed/mok/dozenos-dev-2025-linux.pem"
...
find /lib/modules -type f -name \*.ko | while read MODULE; do
    modinfo ${MODULE} | grep -q "signer:"
    if [ $? != 0 ]; then
        echo "E: Module ${MODULE} is not signed!"
        ...
    fi
done

if [ ! -f ${KERNEL_KEY} ] && [ ! -f ${KERNEL_CERT} ]; then
    echo "I: Signing key for Linux Kernel not found - Secure Boot not possible"
else
    sbsign --key ${KERNEL_KEY} --cert ${KERNEL_CERT} /boot/${VMLINUZ} --output /boot/${VMLINUZ}
    sbverify --list /boot/${VMLINUZ}
    rm -f ${KERNEL_KEY}
fi
```

Three distinct things happen here, in order:

1. **Module-signature verification (glob, not signing).** `find /lib/modules
   -name '*.ko'` globs every installed `.ko` and checks (`modinfo | grep
   signer:`) that it was already signed — by the mechanism in §2, at
   package-build time. This loop does **not itself sign anything** (the
   `sign-file` binary is not invoked here). It is a late sanity check that
   prints a warning and pauses for a keypress on failure, but does **not**
   abort the build — a module that slipped through unsigned would not hard-fail
   the ISO build. That is stock upstream behavior; flagged here as a known
   soft spot, not something item #10 changes.
2. **Kernel EFI signing (the actual MOK step).** If `KERNEL_KEY`/`KERNEL_CERT`
   are both present, `sbsign` signs `/boot/vmlinuz` in place with the MOK,
   then `sbverify --list` prints the embedded signature for the build log.
3. **Guard: bail out if a private key would ship.** After signing, the
   hook `rm -f`s the in-chroot private key, then scans every remaining file
   under `/var/lib/shim-signed/mok/` for a `BEGIN PRIVATE KEY` marker and
   `exit 1`s the whole hook (failing the build) if one is found. This is the
   safety net that stops a private key from ever landing in a shipped image,
   even if the `rm -f` above were somehow skipped.

**Key-path debrand, confirmed reproduced under mode B** (fresh clone →
`rename-transform.sh` → strip `.github/` → `wire-prebuild-hooks.sh` +
`apply-overlay.sh --ci`, per `mirror-push.sh --build-repo`), quoted directly
from the reproduced tree's
`data/live-build-config/hooks/live/93-sb-sign-kernel.chroot`:

```
KERNEL_KEY="/var/lib/shim-signed/mok/dozenos-dev-2025-linux.key"
KERNEL_CERT="/var/lib/shim-signed/mok/dozenos-dev-2025-linux.pem"
```

`dozenos-dev-2025-linux` — not `vyos-dev-2025-linux` — confirming the
four-form transform's automatic rename covers this file's content (not just
its own filename) with no hand-fix needed in the mirrored tree.

**Local (no key) vs CI (signed) contrast:** `data/certificates/` ships empty
(only `README.md` + `.gitignore`, both from `overlay/new-files/`) in every
fresh clone/transform and in any local build where `inject-mok-cert.sh` is
never invoked. `build-dozenos-image` still copies that empty directory into
`includes.chroot/var/lib/shim-signed/mok/`, so the hook's
`[ ! -f ${KERNEL_KEY} ] && [ ! -f ${KERNEL_CERT} ]` guard is true, it prints
"Signing key for Linux Kernel not found - Secure Boot not possible", and the
build **continues and succeeds** — an unsigned kernel that boots fine with
Secure Boot off, which is the local/dev default. Nothing about this flow
requires the local builder to have (or fake) a key.

## 4. Naming reconciliation (resolved)

`data/certificates/README.md` (as shipped pre-#10) flagged an open question:
the signing hook reads a `*-dev-*`-named keypair
(`dozenos-dev-2025-linux.{key,pem}`), while the earlier plan assumed the
CI-injected **enrollment** cert would be `*-prod-*`-named
(`dozenos-prod-2025-linux.pem`, mirroring upstream's pre-rebrand convention
and the filename `overlay/value-fixes/remove-committed-mok-cert.sh` deletes
from a fresh clone). Left as two different names, that mismatch would mean
shim enrolls a cert that does not correspond to the key that signed the
kernel — Secure Boot verification would fail.

**Resolution, implemented by `release/inject-mok-cert.sh`:** there is only
ever **one** injected keypair, and it is written under the **`*-dev-*`**
name for *both* halves:

```
data/certificates/dozenos-dev-2025-linux.key   (private, mode 0600, gitignored)
data/certificates/dozenos-dev-2025-linux.pem   (public,  mode 0644)
```

Because `build-dozenos-image` copies the whole `data/certificates/`
directory verbatim (filenames preserved) into
`includes.chroot/var/lib/shim-signed/mok/`, the same `.pem` this script
writes serves **both** roles at once: it is the cert `sbsign` signs
`vmlinuz` against, *and* it is the cert that ships in the image for
shim/MOK enrollment. One name, one file, no `*-prod-*` copy needed — the
`*-prod-*` naming convention from the pre-#10 plan is superseded by this
decision. `data/certificates/README.md`'s "Naming reconciliation TODO"
section is updated to point here.

## 5. End-to-end flow

```
 GitHub org secrets                    dozenos-build checkout (CI job)
 ───────────────────                   ───────────────────────────────
 MOK_SIGNING_KEY   (base64) ──┐
 MOK_SIGNING_CERT  (base64) ──┼──► release/inject-mok-cert.sh <tree>
                               │        writes:
                               │        data/certificates/dozenos-dev-2025-linux.key  (0600)
                               │        data/certificates/dozenos-dev-2025-linux.pem  (0644)
                               │
                               ▼
                    build-dozenos-image (scripts/image-build/build-dozenos-image:369-371)
                        shutil.copytree('data/certificates',
                            '<lb_config_dir>/includes.chroot/var/lib/shim-signed/mok')
                               │
                               ▼
                    live-build chroot assembly stages includes.chroot/ into the
                    chroot root -> /var/lib/shim-signed/mok/dozenos-dev-2025-linux.{key,pem}
                               │
                               ▼
                    93-sb-sign-kernel.chroot runs (near end of chroot hooks):
                        - verifies every installed .ko already has a "signer:" (§2/§3.1)
                        - sbsign --key ...key --cert ...pem /boot/vmlinuz --output /boot/vmlinuz
                        - sbverify --list /boot/vmlinuz   (build-log proof of signature)
                        - rm -f .../dozenos-dev-2025-linux.key   (private key never ships)
                        - bail (exit 1) if any *.key-shaped private key remains under mok/
                               │
                               ▼
                    ISO assembled with:
                        - a MOK-signed /boot/vmlinuz
                        - /var/lib/shim-signed/mok/dozenos-dev-2025-linux.pem (public enrollment cert)
                        - shim (item #11, deferred/unbuilt — see missing.md row 14) would chain-load
                          this signed kernel once its own Microsoft-countersigned shim + MOK
                          enrollment is in place
                               │
                               ▼
                    item #13 "Signed SB-capable ISO" consumes this: the nightly/
                    integration ISO build now produces a Secure-Boot-capable image
                    instead of the SB-off image the local build produced
```

## 6. Where the CI injection step is wired

`overlay/new-files/.github/workflows/package-smoketest.yml`'s `build_iso`
job (item #8) already runs a full `build-dozenos-image` inside Docker. A
step calling `release/inject-mok-cert.sh` is added **immediately before**
the "Build custom ISO image" step, guarded so it is a no-op when the
secrets are not configured (e.g. a fork, or before the org secrets existed):

```yaml
- name: Inject MOK signing key+cert
  if: ${{ secrets.MOK_SIGNING_KEY != '' && secrets.MOK_SIGNING_CERT != '' }}
  env:
    MOK_SIGNING_KEY: ${{ secrets.MOK_SIGNING_KEY }}
    MOK_SIGNING_CERT: ${{ secrets.MOK_SIGNING_CERT }}
  run: dozenos-rebrand/release/inject-mok-cert.sh dozenos-build
```

placed before `docker run ... build-dozenos-image` so the keypair is on
disk at `dozenos-build/data/certificates/` by the time `build-dozenos-image`
runs its `copytree` (§5). A matching cleanup step runs after the build
regardless of outcome, shrinking the window the decoded private key sits in
the job's own git worktree (the runner is destroyed at job end either way,
so this is defense in depth, not the only removal — `93-sb-sign-kernel.chroot`
itself already removes its own in-chroot copy, see §3):

```yaml
- name: Clean up injected MOK key
  if: ${{ always() && secrets.MOK_SIGNING_KEY != '' }}
  run: dozenos-rebrand/release/inject-mok-cert.sh dozenos-build --cleanup
```

**Why here and not only item #13:** `package-smoketest.yml` is the only
workflow that currently runs a real `build-dozenos-image` end-to-end
(item #17's nightly-build workflow, item #13's actual deliverable, is not
authored yet — see `.powerloop/2026-07-07-cicd.note.md` item #13: "pending").
Wiring the guarded step here means SB signing is exercised (when secrets are
present) on every push to `rolling`, not just once nightly builds exist —
and the guard means nothing changes for this workflow when the secrets are
absent (identical to today's unsigned behavior, §3 "Local vs CI"). Item #13's
nightly workflow, once authored, should use the **same two steps** verbatim
(same script, same guard) — this document is the reference for that
wiring, so item #13 does not have to re-derive it.

## 7. Post-build verification (maintainer checklist)

Run against a CI-built (or any real-key-signed) ISO/chroot, not a local
SB-off build:

- **Kernel signature present:**
  `sbverify --cert data/certificates/dozenos-dev-2025-linux.pem /boot/vmlinuz`
  (or extract `vmlinuz` from the built ISO first) — expect
  `Signature verification OK`. The build log itself already contains one
  `sbverify --list` run from step 2 in §3; this is the same check re-run
  independently against the finished artifact.
- **Module signatures present:** `modinfo <module>.ko | grep -A2 signer:`
  for a sample from each recipe in §2's table — expect a `signer:`,
  `sig_hashalgo:`, and `sig_key:` field, matching the kernel's own
  `CONFIG_MODULE_SIG_HASH="sha512"` (§ kernel config,
  `scripts/package-build/linux-kernel/config/02-module-signing.config`).
- **No private key shipped:** `grep -rl "BEGIN.*PRIVATE KEY"` across the
  built chroot/ISO contents should find nothing. This is also what
  `93-sb-sign-kernel.chroot`'s own guard (§3.3) already enforces at build
  time — a clean build log with no "Found private key - bailing out" line
  is the first signal; the grep is the independent re-check.
- **MOK enrollment (item #11, `shim-signed` — see §9 below):** on real
  hardware/VM with Secure Boot on, run the `install mok` op-mode command
  (`install_mok.sh`, shipped by dozenos-1x), which runs `mokutil
  --ignore-keyring --import /var/lib/shim-signed/mok/dozenos-dev-2025-shim.der`
  — **note DER, not PEM**: `mokutil --import` requires a DER-encoded cert,
  which is why this is a separate file from the `.pem` §3/§5 sign the kernel
  with, not the same file re-read. This stages a pending enrollment; the
  `MokManager` first-boot UI then prompts to commit it into the firmware's
  MOK list. `mokutil --list-enrolled` afterward should show the DozenOS MOK
  subject. See §9 for the full shim/MOK-enrollment flow.

## 8. Explicit non-goals of this document / item #10

- Does **not** get the MOK cert countersigned by the Microsoft UEFI CA —
  that is a separate, non-technical process (`DOZENOS-CICD-PLAN.md` row 3);
  this document assumes a self-enrolled MOK, the same trust model VyOS
  itself ships.
- Does **not** generate, rotate, or handle the real key. `inject-mok-cert.sh`
  only ever reads the secret **names**; no key material appears in this
  repo, this document, or any script under `dozenos-rebrand/`.
- `shim-signed` itself (item #11) is covered in §9, not here — §1-§8 above
  describe item #10 (kernel/module signing) only.

## 9. `shim-signed` / MOK enrollment (item #11)

### 9.1 What `scripts/package-build/shim-signed/` actually produces

`package.toml` clones `github.com/dozenos/shim-signed.git` (the mirrored,
rename-transformed fork of Debian's `shim-signed` **source** package —
Vcs-Git points at `salsa.debian.org/efi-team/shim-signed`) and runs a plain
`dpkg-buildpackage` via the generic `build.py` driver. There is no
`build_cmd` override and no cert-consuming pre/post-build logic in
`package.toml` beyond the generic
`pre_build_hook = "/dozenos-rebrand/rename-transform.sh ."` every recipe
gets (debrands package name/metadata/maintainer strings in the cloned tree,
nothing cert-related).

Confirmed by reading the actual mirrored source (`git clone --depth 1
https://github.com/vyos/shim-signed.git`, read-only, scratch dir only):
the repo ships **pre-built, already Microsoft-signed** shim binaries
committed as binary blobs —
`shimx64.efi.signed.MS-2011` / `shimx64.efi.signed.MS-2023` (~1 MB each) —
plus test-only `TEST-shimx64.efi.signed.{multi,snakeoil}`. `debian/control`
describes the `shim-signed` binary package as: *"This package contains the
version of the bootloader binary signed by the Microsoft UEFI CA(s)."*
`debian/rules` is a stock `dh $@` wrapper with no step that builds shim
from source or invokes any signing tool — it packages the already-signed
binaries plus `shim-helpers-amd64-signed`/`shim-unsigned` (Debian
Build-Depends) into installable `.deb`s.

**Conclusion — this is case (a), not (b) or (c):** the recipe **repackages
Debian's Microsoft-signed shim binary as-is**. It does not compile shim from
source, and no `VENDOR_CERT`/vendor cert is embedded into shim during this
build. This cannot be otherwise: re-signing shim with a DozenOS-specific
vendor cert requires submitting that cert to Microsoft's own UEFI CA
signing process (a manual, non-technical, out-of-band process — already
named as out of scope in §8/`DOZENOS-CICD-PLAN.md` row 3). No CI secret or
script wiring can substitute for that. `shim-signed`'s own scripts
(`update-secureboot-policy`, `debian/source_shim-signed.py`) only reference
DKMS module-signing MOK state (`/var/lib/dkms/mok.pub`,
`MokSB-*`/`MokSBStateRT-*` EFI vars) — **not**
`/var/lib/shim-signed/mok/`, which is a DozenOS/VyOS-specific staging
convention (§3/§5) that `shim-signed`'s own Debian packaging neither reads
nor writes.

**Correcting the earlier item framing:** `missing.md` row 14 described the
gap as "`shim-signed` NOT built... requires a Secure Boot signing
certificate." That is imprecise: building `shim-signed` itself needs **no
cert at all** (§9.1 above) — `rebuild-packages.yml` has zero
cert/shim/MOK references, confirming the package build is cert-free. The
actual gap is downstream: the **shipped image** needs the DozenOS MOK cert
in DER form so `install_mok.sh` can enroll it (§9.2). Local builds skipping
`shim-signed` was still the right call operationally (Debian's own
`shim-signed`, unmodified, satisfies the `dozenos-1x` Depends and boots
fine with Secure Boot off), just not for the reason originally recorded.

### 9.2 The real trust flow

```
 Microsoft UEFI CA (out of band, not DozenOS's)
      │  signs shim binary once, ships in Debian's shim-helpers-*-signed
      ▼
 shimx64.efi.signed  (repackaged verbatim by dozenos/shim-signed, §9.1)
      │  firmware trusts this via UEFI db (Microsoft-signed)
      ▼
 shim, at boot: tries to verify next-stage loader (grubx64.efi / vmlinuz)
      │
      ├─ signed by a cert shim already trusts (its own baked-in vendor
      │  cert, or the UEFI db)?  DozenOS's kernel is NOT signed by either
      │  of those (§1: it's signed by the MOK, an independent keypair) —
      │  so this path does not apply to DozenOS.
      │
      └─ NOT trusted by the above → shim consults the enrolled MOK list
         (MokListRT, populated by a prior `mokutil`/`MokManager` run) →
         DozenOS's MOK cert (dozenos-dev-2025-linux.pem, ENROLLED as
         dozenos-dev-2025-shim.der) is on that list → kernel signature
         (§1/§3, sbsign with the SAME MOK) verifies against it → boot
         proceeds.
```

**The DozenOS-specific part of Secure Boot is therefore NOT "building
shim-signed with the SB cert"** — shim is never re-signed by or for
DozenOS. It is:

1. Debranding `shim-signed`'s package name/metadata (generic
   `rename-transform.sh` pre_build_hook, §9.1 — already covered by the
   pre-existing per-recipe wiring, nothing new needed for item #11).
2. Ensuring the DozenOS MOK cert — the same one §1/§3/§5 use to sign
   `vmlinuz` — is what gets **enrolled** into the firmware's MOK list, in
   the exact encoding (`install_mok.sh` / `mokutil --import` needs DER,
   not PEM) and exact filename it expects.

### 9.3 Cert materialization: `dozenos-dev-2025-shim.der`

`data/certificates/README.md` documents this from the shipped-tree side;
this is the toolkit-side record. `../release/inject-mok-cert.sh` (extended
for item #11) now also derives the DER re-encoding, in the SAME write-mode
invocation that writes the `.key`/`.pem` pair, from the SAME `.pem` it just
wrote — never a second/different cert:

```
openssl x509 -in data/certificates/dozenos-dev-2025-linux.pem \
             -outform DER -out data/certificates/dozenos-dev-2025-shim.der
```

Fails loudly (exits non-zero) if `openssl` is missing, the conversion
fails, or the resulting file is empty — same fail-loud posture as the rest
of the script, no silent fallback. `--cleanup` removes the `.der` alongside
the `.key`/`.pem`, idempotently. Filename is not a free choice: it is
exactly what `install_mok.sh` (a **pristine upstream file** — the
mirror/rename-transform pipeline automatically rewrites its
`vyos-dev-2025-shim.der` reference to `dozenos-dev-2025-shim.der`, verified
by diffing the four-form transform's output against upstream HEAD; no
hand-fix or naming decision was needed here, unlike item #10's dev/prod
reconciliation) expects at `/var/lib/shim-signed/mok/dozenos-dev-2025-shim.der`.

Because `build-dozenos-image` copies `data/certificates/` **wholesale**
(§5) into `includes.chroot/var/lib/shim-signed/mok/`, adding the `.der`
required zero changes to `build-dozenos-image`, `93-sb-sign-kernel.chroot`,
or any live-build hook — it rides the same `copytree` item #10 already
wired.

**Cert-chain consistency (verified statically):** `.pem` and `.der` are
provably the same certificate — the DER is a pure re-encoding of the exact
`.pem` bytes just written, not derived from `MOK_SIGNING_CERT` a second
time and not a separately-sourced file. `test/test-inject-mok-cert.sh`
codifies this: it computes `openssl x509 ... -fingerprint -sha256` for both
the `.pem` and the `.der` (with `-inform DER`) and asserts they are
byte-identical. A throwaway self-signed test cert (mktemp-only, `CN=
THROWAWAY TEST ONLY`, never committed) was used to exercise this — see the
script; no real key was generated or handled to verify this document.

### 9.4 `.gitignore` / tree cleanliness

`data/certificates/.gitignore` stays `*.key` — unchanged. The `.der`, like
the `.pem` it is derived from, is **public** material and is deliberately
**not** gitignored, for the same reason the `.pem` already is not: both are
written by `inject-mok-cert.sh` immediately before a build and removed by
its `--cleanup` mode immediately after (§6/CI wiring below), so there is no
window where an accidental `git add -A` in a CI job's own worktree would
pick either up without also picking up the (gitignored) `.key` — and no
local developer is expected to run `inject-mok-cert.sh` against a tree they
intend to `git add -A` in. Gitignoring only the `.der` while leaving the
`.pem` un-gitignored would be an inconsistency with no functional benefit,
so it was deliberately not added.

### 9.5 CI wiring

No new CI step. `shim-signed` is built as an ordinary C2 package by
`rebuild-packages.yml`, at package-build time — and, per §9.1, that build
never touches a cert. The `.der` is a **shipped-image-time** artifact
(consumed by `install_mok.sh` only after the ISO boots, on a real machine
with Secure Boot hardware — nothing in the build or test pipeline runs
`mokutil --import` itself), so it only needs to exist wherever
`data/certificates/` is populated before `build-dozenos-image`'s copytree
runs — which is exactly where item #10 already calls
`release/inject-mok-cert.sh`, in `package-smoketest.yml`'s `build_iso` job
(§6). Since the `.der` derivation was added **inside** `inject-mok-cert.sh`
itself (not a sibling script), the existing guarded
`if: secrets.MOK_SIGNING_KEY != '' && secrets.MOK_SIGNING_CERT != ''` step
and its matching `--cleanup` step now produce/remove all three files with
no workflow YAML change required. Item #13's nightly workflow, once
authored, inherits this for free by following §6's guidance to reuse the
same two steps verbatim.

### 9.6 What is statically verified vs. CI-only

**Statically verified in this cycle** (mode-B mirror reproduction +
mirrored-source read + throwaway-cert script run, all read-only/scratch,
no real key, no real build):
- `shim-signed`'s recipe and actual mirrored source content (repackage,
  not compile/sign) — §9.1.
- `install_mok.sh`'s exact expected filename or/DER requirement, and that
  it is a pristine, correctly-transformed upstream file — §9.3.
- `inject-mok-cert.sh`'s DER derivation: fail-loud, idempotent, correct
  mode/filename, PEM≡DER fingerprint match — exercised end-to-end against a
  throwaway self-signed cert, and covered by
  `test/test-inject-mok-cert.sh` (20 assertions, all passing).
- That no workflow currently runs `mokutil --import`/enrolls a MOK at all
  (grep across `.github/workflows/` in the reproduced mirror) — enrollment
  is inherently a real-hardware/VM-firmware, post-boot, human/first-boot
  step, not something CI does.

**CI-only / cannot be verified without the real key + a real build:**
- That a CI build with the real `MOK_SIGNING_KEY`/`MOK_SIGNING_CERT`
  actually produces a `.der` whose fingerprint matches the real MOK (this
  document proves the *mechanism* is fingerprint-preserving; it does not
  and cannot exercise it with the real secret).
- That Debian's `shim-helpers-amd64-signed` dependency resolves cleanly in
  the real build container (package-build-time apt resolution, not
  something this audit's static/read-only pass exercises).
- The actual `mokutil --import` + `MokManager` enrollment + a
  Secure-Boot-on boot test against a real signed kernel — real hardware/VM
  + real firmware + real key, entirely out of scope for a toolkit-repo
  audit. Tracked as item #13's eventual boot-test responsibility (§7's
  checklist already names this).
