### GPG role reconciliation (no apt mirror)

The original plan (`DOZENOS-CICD-PLAN.md` Phase 1 row 1) described
`GPG_PRIVATE_KEY` as signing "the DozenOS apt repo `Release`/`InRelease` so
apt trusts it." **That no longer applies.** The architecture decision
(cicd note item 9) is:

- DozenOS is **image-based upgrade** (swap ISO, like VyOS) ŌĆö there is **no
  runtime `apt upgrade`** and **no public/persistent apt mirror**.
- Inside a build job, packages are served from an **ephemeral, in-job apt
  repo** (`dpkg-scanpackages`/`reprepro` over the deb-store, pointed to by
  `--vyos-mirror file://...` or `localhost`) with **`[trusted=yes]`** ŌĆö it
  vanishes when the runner exits, so **GPG-signing that ephemeral repo's
  `Release` file is not required.**

So `GPG_PRIVATE_KEY`'s actual remaining role is narrower than the plan's
original wording:

1. Signs the **GitHub Release** assets / build artifacts published by
   `dozenos-nightly-build` (detached `.asc` signature alongside the ISO and
   `.minisig`, if/when a workflow chooses to add one).
2. Its public half ships in the ISO as the **archive keyring** (the trust
   anchor a future consumer would use to verify DozenOS-signed artifacts),
   not as an apt `Release`-signing key for a live mirror.

If a persistent apt mirror is ever reintroduced, `GPG_PRIVATE_KEY`'s
Release-signing role from the original plan is restored as-is; until then,
treat it as an artifact/Release signing key, not an apt-trust key.

## 2. Org-level setup notes

- **Where:** `github.com/organizations/dozenos/settings/secrets/actions` ŌĆö
  Repository secrets tab has an org-level "Secrets" and "Variables" section.
- **Scope:** *Selected repositories* ŌĆö `dozenos-build`, `dozenos-nightly-build`,
  and any signing/package repo that needs them. Do **not** use "All
  repositories" (least-privilege: a repo that doesn't build/sign artifacts
  has no reason to see these secrets).
- **Private key files ŌåÆ base64:** every private-key-bearing secret
  (`GPG_PRIVATE_KEY`, `MINISIGN_SECRET_KEY`, `MOK_SIGNING_KEY`,
  `MOK_SIGNING_CERT`) is stored **base64-encoded with no line wraps**:
  `base64 -w0 <file>` ŌĆö paste that single-line output as the secret value.
- **Decode in-workflow**, write to a runner-local path, restrict
  permissions, and remove after use:
  ```yaml
  - name: Decode signing key
    run: |
      umask 077
      echo "${{ secrets.MOK_SIGNING_KEY }}" | base64 -d > "$RUNNER_TEMP/mok.key"
      chmod 600 "$RUNNER_TEMP/mok.key"
      # ... use it ...
      rm -f "$RUNNER_TEMP/mok.key"
  ```
  Never `echo`/`cat` the decoded file contents to the log ŌĆö GitHub Actions
  masks the *secret value itself* in log output, but a decoded intermediate
  form (e.g. re-encoded, reformatted, or partially printed) is not
  automatically masked. Write to `$RUNNER_TEMP` (wiped when the job ends),
  not the workspace, and `rm -f` explicitly once done.

  (This snippet is a generic illustration for org-secret handling in
  general. The actual `MOK_SIGNING_KEY`/`MOK_SIGNING_CERT` consumer,
  `release/inject-mok-cert.sh`, necessarily writes into the workspace ŌĆö
  `<checkout>/data/certificates/` ŌĆö rather than `$RUNNER_TEMP`, because
  `build-dozenos-image` reads that exact repo-relative path at build time;
  it still `chmod 600`s the key and the workflow removes it with
  `inject-mok-cert.sh --cleanup` once the build step finishes. See
  `SB-SIGNING.md` for the real flow.)
- **Repo-level override risk:** a **repository-level** secret with the same
  name **silently takes precedence** over the org-level one for that repo.
  When auditing, check each selected repo's own Settings ŌåÆ Secrets for a
  same-named override before assuming the org value is what actually runs ŌĆö
  an accidental or stale repo-level secret is a way this checklist could
  pass while the workflow still uses different key material.
- **Rotation:** the GPG key should carry its own expiry (already the plan's
  `gpg --full-generate-key` guidance); the PAT is the highest-churn
  credential here ŌĆö see the deferred migration in ┬¦4, and rotate it
  immediately if it is ever exposed in a log or a fork PR context.

## 5. `UPSTREAM_URL` ŌĆö per-repo secret (item #14 self-sync)

Every other secret/variable in this doc is **org-level**, scoped to
*Selected repositories* (┬¦2). `UPSTREAM_URL` is the one deliberate exception:
it is a **repository-level** secret, set **individually, once per mirror**,
on all 17 `dozenos/*` mirrors ŌĆö `github.com/dozenos/<name>/settings/secrets/actions`,
not the org settings page.

- **Value:** the plain upstream URL that mirror resyncs from, e.g.
  `https://github.com/vyos/hvinfo` for `dozenos/hvinfo`,
  `https://github.com/vyos/vyos-build` for `dozenos/dozenos-build`,
  `https://github.com/vyos/vyos-1x` for `dozenos/dozenos-1x`. Not base64 ŌĆö
  a plain URL string, same format `mirror-push.sh`'s own `<upstream-url>`
  positional argument expects.
- **Why per-repo, not org-level:** an org-level secret has exactly one
  value shared by every repo it's scoped to ŌĆö useless here, since each of
  the 17 mirrors needs a *different* upstream URL. A repo-level secret with
  the same name as an org-level one would silently override it anyway (see
  ┬¦2's "Repo-level override risk" note) ŌĆö `UPSTREAM_URL` simply has no
  org-level counterpart to override, by design.
- **The only remaining `vyos` residual, and only at CI runtime.** This
  mapping is the one place a `github.com/vyos/*` URL is allowed to exist in
  this entire toolkit ŌĆö and even here, it lives only inside a GitHub secret
  value, never inside `mirror-push.sh`, `sync.yml.template`, or any
  generated `sync.yml` (which only ever reference
  `${{ secrets.UPSTREAM_URL }}`, not the literal URL). GitHub's own secret
  masking redacts the value from workflow logs once registered.
- **Consumed by:** the generated `.github/workflows/sync.yml` on each
  mirror (item #14) ŌĆö see `SYNC.md` for the full workflow spec, and
  `sync.yml.template` / `mirror-push.sh`'s `generate_sync_workflow()` for
  how that file is produced. The workflow's first real step fails loud
  (`::error::`, `exit 1`) if this secret is unset on a given repo, rather
  than letting `mirror-push.sh` fail confusingly on a missing/empty
  upstream-URL argument.
- **Not yet set anywhere** ŌĆö this is a specification for the eventual
  roll-out (blocked on item #20 + this secret being injected on all 17
  repos, see `SYNC.md` ┬¦6), not a claim that it has been injected. The
  assistant does not generate or request URLs on the user's behalf beyond
  what is already public knowledge (the `github.com/vyos/<name>` mapping is
  the whole point of this toolkit and appears throughout it as a CLI
  argument already); no private key material is involved in this secret at
  all.
