# DozenOS CI — Org Secrets & Variables Checklist

Phase 1 of `DOZENOS-CICD-PLAN.md`. The user has **already generated and
injected** every key below as a GitHub Actions **organization-level** secret
at `github.com/organizations/dozenos/settings/secrets/actions`. The assistant
never generates, requests, or handles private key material — this doc only
records the exact **names**, **stored formats**, and **which CI build step
consumes each one**, so the injected names can be verified against the plan
and wired into workflows by reference (`secrets.<NAME>`).

No key bytes, fingerprints, or passphrases appear anywhere in this file.

### GPG role reconciliation (no apt mirror)

The original plan (`DOZENOS-CICD-PLAN.md` Phase 1 row 1) described
`GPG_PRIVATE_KEY` as signing "the DozenOS apt repo `Release`/`InRelease` so
apt trusts it." **That no longer applies.** The architecture decision
(cicd note item 9) is:

- DozenOS is **image-based upgrade** (swap ISO, like VyOS) — there is **no
  runtime `apt upgrade`** and **no public/persistent apt mirror**.
- Inside a build job, packages are served from an **ephemeral, in-job apt
  repo** (`dpkg-scanpackages`/`reprepro` over the deb-store, pointed to by
  `--vyos-mirror file://...` or `localhost`) with **`[trusted=yes]`** — it
  vanishes when the runner exits, so **GPG-signing that ephemeral repo's
  `Release` file is not required.**

So `GPG_PRIVATE_KEY`'s actual remaining role is narrower than the plan's
original wording:

1. Signs the **GitHub Release** assets / build artifacts published by
   `dozenos-nightly-build` (detached `.asc` signature alongside the ISO and
   `.minisig`, if/when a workflow chooses to add one).
2. Its public half ships in the ISO as the **archive keyring** (the trust
   anchor a future consumer would use to verify DozenOS-signed artifacts),
   not as an apt `Release`-signing key for a live mirror.

If a persistent apt mirror is ever reintroduced, `GPG_PRIVATE_KEY`'s
Release-signing role from the original plan is restored as-is; until then,
treat it as an artifact/Release signing key, not an apt-trust key.

## 2. Org-level setup notes

- **Where:** `github.com/organizations/dozenos/settings/secrets/actions` —
  Repository secrets tab has an org-level "Secrets" and "Variables" section.
- **Scope:** *Selected repositories* — `dozenos-build`, `dozenos-nightly-build`,
  and any signing/package repo that needs them. Do **not** use "All
  repositories" (least-privilege: a repo that doesn't build/sign artifacts
  has no reason to see these secrets).
- **Private key files → base64:** every private-key-bearing secret
  (`GPG_PRIVATE_KEY`, `MINISIGN_SECRET_KEY`, `MOK_SIGNING_KEY`,
  `MOK_SIGNING_CERT`) is stored **base64-encoded with no line wraps**:
  `base64 -w0 <file>` — paste that single-line output as the secret value.
- **Decode in-workflow**, write to a runner-local path, restrict
  permissions, and remove after use:
  ```yaml
  - name: Decode signing key
    run: |
      umask 077
      echo "${{ secrets.MOK_SIGNING_KEY }}" | base64 -d > "$RUNNER_TEMP/mok.key"
      chmod 600 "$RUNNER_TEMP/mok.key"
      # ... use it ...
      rm -f "$RUNNER_TEMP/mok.key"
  ```
  Never `echo`/`cat` the decoded file contents to the log — GitHub Actions
  masks the *secret value itself* in log output, but a decoded intermediate
  form (e.g. re-encoded, reformatted, or partially printed) is not
  automatically masked. Write to `$RUNNER_TEMP` (wiped when the job ends),
  not the workspace, and `rm -f` explicitly once done.

  (This snippet is a generic illustration for org-secret handling in
  general. The actual `MOK_SIGNING_KEY`/`MOK_SIGNING_CERT` consumer,
  `release/inject-mok-cert.sh`, necessarily writes into the workspace —
  `<checkout>/data/certificates/` — rather than `$RUNNER_TEMP`, because
  `build-dozenos-image` reads that exact repo-relative path at build time;
  it still `chmod 600`s the key and the workflow removes it with
  `inject-mok-cert.sh --cleanup` once the build step finishes. See
  `SB-SIGNING.md` for the real flow.)
- **Repo-level override risk:** a **repository-level** secret with the same
  name **silently takes precedence** over the org-level one for that repo.
  When auditing, check each selected repo's own Settings → Secrets for a
  same-named override before assuming the org value is what actually runs —
  an accidental or stale repo-level secret is a way this checklist could
  pass while the workflow still uses different key material.
- **Rotation:** the GPG key should carry its own expiry (already the plan's
  `gpg --full-generate-key` guidance). The former highest-churn credential,
  `BUILD_PAT`, is gone (§4): cross-repo tokens are now minted per-job at
  runtime and expire in ~1h on their own. The remaining long-lived credential
  is `BUILD_APP_PRIVATE_KEY` — if it is ever exposed, revoke+regenerate the
  key from the App's settings page (old key invalidates immediately) and
  paste the new PEM over the org secret.

## 4. `BUILD_PAT` → GitHub App migration (DONE 2026-07-09)

Originally deferred per `DOZENOS-CICD-PLAN.md` Phase 1 (user decision
2026-07-07), executed 2026-07-09. The fine-grained PAT was tied to one
personal account; the org-owned **GitHub App** replaces it with per-job,
~1h-lived installation tokens (higher rate limits, per-repo scoping at
mint time). What was done:

1. User created the org GitHub App: permissions Contents: Read/Write +
   Actions: Read/Write; no webhook, no OAuth/user-authorization anything
   (only server-to-server installation tokens are ever used, so the
   "Expire user authorization tokens" setting is irrelevant); installed
   on the `dozenos` org.
2. User injected org **Variable** `BUILD_APP_ID` + org **Secret**
   `BUILD_APP_PRIVATE_KEY` (raw PEM — deliberately NOT base64, deviating
   from the original plan text: `actions/create-github-app-token` consumes
   PEM directly).
3. Every `secrets.BUILD_PAT` usage in the workflows was replaced with a
   token minted at the start of the consuming job via
   `actions/create-github-app-token@v3`, `repositories:` narrowed to
   exactly what that job touches (the one exception: nightly's
   check-changes mints org-wide, since its whole job is enumerating the
   mirror set). Touched: `sync.yml.template` (dispatch step),
   `overlay-dozenos-build/new-files/.github/workflows/{rebuild-dispatch,
   rebuild-packages}.yml`, and
   `dozenos-nightly-build`'s `nightly.yml`. Token lifetime (~1h)
   vs. multi-hour builds is a non-issue by construction: every job uses
   its token only in its opening minutes (checkouts / API reads), never
   after the long build steps — `nightly.yml`'s build-image job carries a
   comment enforcing that invariant for future edits.
4. **The PAT was revoked and the `BUILD_PAT` org secret deleted** (user,
   2026-07-09) — this happened BEFORE the workflow changes were pushed, so
   the window between deletion and roll-out is a known
   cross-repo-CI-is-broken window, closed by the roll-out itself.

⚠️ **Residual tightening (deliberate, temporary):** the App installation
AND both `BUILD_APP_ID`/`BUILD_APP_PRIVATE_KEY` were set to
**all-repositories access** for bring-up speed. Least-privilege follow-up:
narrow the secret/variable to *Selected repositories* = the 20
token-minting repos (19 mirrors + `dozenos-nightly-build`). The App
*installation* itself can stay org-wide — check-changes' enumeration
depends on it, and installation grants nothing to a repo's own workflows
without the private key.

**Workflows: Read/Write permission (GRANTED by user 2026-07-09, verified
via `gh api orgs/dozenos/installations` → `permissions.workflows:
"write"`):** the mirror self-push moved off `github.token` onto an
App-minted token (`sync.yml.template`'s "Mint GitHub App token
(self-push)" step) because `GITHUB_TOKEN` categorically cannot create or
update `.github/workflows/*` files — first hit 2026-07-09 when the
deb-cache change to `rebuild-dispatch.yml` made `dozenos-build`'s
self-sync push a changed workflow file (`! [remote rejected] ... without
"workflows" permission`); every earlier sync passed only because the
regenerated workflow files were byte-identical. The App token inherits
the App's permissions, so the App additionally grants **Workflows: Read
and write** — this is what lets a self-sync propagate template/overlay
workflow-content changes; if this permission is ever revoked, self-syncs
fail again whenever a workflow file's content actually changes
(byte-identical steady-state syncs would still pass).

## 5. `UPSTREAM_URL` — per-repo variable (item #14 self-sync)

Every other secret/variable in this doc is **org-level**, scoped to
*Selected repositories* (§2). `UPSTREAM_URL` is the one deliberate exception:
it is a **repository-level Actions *variable*** (non-secret), set
**individually, once per mirror**, on all 17 `dozenos/*` mirrors —
`github.com/dozenos/<name>/settings/variables/actions`, not the org settings
page. It is a *variable*, not a *secret*, because the value is a plain public
`github.com/vyos/<name>` URL with nothing sensitive in it (see the residual
note below); a secret would only add log-masking noise for no benefit.

- **Value:** the plain upstream URL that mirror resyncs from, e.g.
  `https://github.com/vyos/hvinfo` for `dozenos/hvinfo`,
  `https://github.com/vyos/vyos-build` for `dozenos/dozenos-build`,
  `https://github.com/vyos/vyos-1x` for `dozenos/dozenos-1x`. Not base64 —
  a plain URL string, same format `mirror-push.sh`'s own `<upstream-url>`
  positional argument expects.
- **Why per-repo, not org-level:** an org-level variable has exactly one
  value shared by every repo it's scoped to — useless here, since each of
  the 17 mirrors needs a *different* upstream URL. A repo-level variable with
  the same name as an org-level one would silently override it anyway (see
  §2's "Repo-level override risk" note) — `UPSTREAM_URL` simply has no
  org-level counterpart to override, by design.
- **The only remaining `vyos` residual, and only at CI runtime.** This
  mapping is the one place a `github.com/vyos/*` URL is allowed to exist in
  this entire toolkit — and even here, it lives only inside a GitHub Actions
  variable value, never inside `mirror-push.sh`, `sync.yml.template`, or any
  generated `sync.yml` (which only ever reference
  `${{ vars.UPSTREAM_URL }}`, not the literal URL). The value is a plain
  public URL, so — unlike a secret — GitHub does **not** mask it in workflow
  logs; that is intentional and harmless here (the `vyos/<name>` mapping is
  public knowledge, the whole point of this toolkit).
- **Consumed by:** the generated `.github/workflows/sync.yml` on each
  mirror (item #14) — see `SYNC.md` for the full workflow spec, and
  `sync.yml.template` / `mirror-push.sh`'s `generate_sync_workflow()` for
  how that file is produced. The workflow's first real step fails loud
  (`::error::`, `exit 1`) if this variable is unset on a given repo, rather
  than letting `mirror-push.sh` fail confusingly on a missing/empty
  upstream-URL argument.
- **Set on all 17 mirrors** (rolled out 2026-07-08) as repository Actions
  variables — the earlier secret form was migrated to a variable in the same
  roll-out. No private key material is involved in this value at all.
