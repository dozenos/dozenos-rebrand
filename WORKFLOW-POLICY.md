# WORKFLOW-POLICY.md — DozenOS `.github/workflows/` policy (item #7)

What happens to upstream `vyos-build`'s CI workflows when it becomes
`dozenos/dozenos-build`, and what is and is not allowed to appear in any
DozenOS mirror's `.github/`. This is the durable record for Phase-3 item #7
("delete VyOS-infra workflows"); item #8 (build CI, AUTHORED) and item #14
(self-sync, AUTHORED — see `SYNC.md`) / item #17 (nightly, not yet done)
each own their own workflow *authoring* and are only cross-referenced here,
not duplicated.

## Mechanism: wholesale strip, not selective delete

Mode B never forks-then-deletes individual workflow files. Every mirror push
(`dozenos-rebrand/mirror-push.sh`) removes the ENTIRE upstream `.github/`
directory unconditionally, as step 3/7, before any overlay or build-repo
logic runs, on both `seed` (first push) and `sync` (re-push) modes — it is
plain code in the linear pipeline, not gated by any flag:

```sh
# ------------------------------------------------------------------------- #
# 3) Strip the upstream .github/ entirely (its VyOS-org workflows must never
#    run against our mirror / ping VyOS infra -- see cicd.note item #4).
# ------------------------------------------------------------------------- #
log "3/7 stripping .github/ ..."
rm -rf "$CLONE_DIR/.github"
```

This removes `.github/workflows/*` AND everything else under `.github/`
(`config/`, `mergify.yml`, `PULL_REQUEST_TEMPLATE.md`, `CODEOWNERS` if
present there, etc.) — the whole directory, not just the workflow files.
Because it runs before the seed/sync branch and before `--build-repo`'s
hooks/overlay step, it applies identically to every target repo and every
push mode; there is no code path in `mirror-push.sh` that ships any upstream
`.github/` content.

## Where DozenOS's own workflows come from

The strip only removes; it never adds. The only two sanctioned sources of
`.github/workflows/*` content on any `dozenos/*` mirror are:

(There is a third code path capable of writing files into a clone at all —
the generic per-repo `--overlay <dir>` flag, independent of `--build-repo`.
It is deliberately excluded from this list because it is not permitted to
carry workflow content at all: see "The generic `--overlay <dir>` flag is
excluded from workflows" below for how the engine enforces that.)

- **`dozenos-build`'s own build/rebuild CI (items #8 and #15, AUTHORED
  2026-07-08):** lives under `dozenos-rebrand/overlay-dozenos-build/new-files/.github/workflows/`
  as `build-docker-image.yml`, `rebuild-packages.yml` (item #8) plus
  `rebuild-dispatch.yml` (item #15, the incremental
  rebuild-fan-out receiver for item #14's `repository_dispatch` — see
  `REBUILD-DISPATCH.md`), all copied in by `overlay-dozenos-build/apply-overlay.sh` (the
  same `new-files/` bucket already used for new build recipes — see
  `overlay-dozenos-build/README.md`). Replaces 2 of the build workflows above and adds one
  new one with no upstream counterpart (upstream's `package-smoketest.yml`
  gets no replacement here — ISO build + config-load gate live in
  `dozenos-nightly-build`'s `nightly.yml`); verified to not reintroduce any
  of the 8 infra ones (zero `vyos`, zero `uses: vyos/*`, all secrets/vars
  used are defined in `CI-SECRETS.md`). Full write-up, per-file description, and
  end-to-end repro proof: `overlay-dozenos-build/MANIFEST.md`'s "new-files/ —
  `.github/workflows/` (item #8, LANDED)" and "... `rebuild-dispatch.yml`
  (item #15, LANDED)" sections.
- **Every mirror's `sync.yml` (item #14, AUTHORED 2026-07-08):** unlike
  `dozenos-build`'s build CI above, this is NOT delivered via
  `overlay-dozenos-build/new-files/` — it is generated directly by `mirror-push.sh` itself
  (its `generate_sync_workflow()` step, run for every target) from
  `dozenos-rebrand/sync.yml.template`, right after any `--build-repo`/
  `--overlay` content is applied and before the final `--verify` + push. Same
  sanctioned-source spirit (a controlled, auditable place workflow content
  comes from, never an upstream leftover), different delivery mechanism
  because this one needs per-repo flag substitution (`--build-repo`,
  `--overlay <dir> --allow-residuals`, etc.) that a static overlay file
  can't express. Full design, the per-target baked-flags table, and the
  verified zero-`vyos`/no-`uses: vyos/*` dry-run reproductions: `SYNC.md`.
  `dozenos-build`'s nightly full-image build (item #17, not yet done) is a
  separate, additional workflow on top of `sync.yml`, scoped to
  `dozenos-build` only.

No other mechanism may add `.github/workflows/*` content — `rename-transform.sh`
never authors files, so any workflow found on a mirror necessarily came
either from an upstream repo it wasn't stripped from (a pipeline bug) or
from one of the two sources above.

### The generic `--overlay <dir>` flag is excluded from workflows

`mirror-push.sh` has a separate, generic `--overlay <dir>` flag (independent
of `--build-repo`) for per-repo value-fix content — e.g. `overlay-dozenos-1x/`
for the `dozenos-1x` mirror (default-password hash regeneration, reverting
2 non-mirrored-org refs; see that overlay's own `README.md`). When the given
`<dir>` ships its own executable `apply-overlay.sh`, that script runs and is
trusted to behave (as `overlay-dozenos-1x/apply-overlay.sh` does: value-fix
scripts only, never touches `.github/`). When it does not, `mirror-push.sh`
falls back to a plain recursive copy of the directory's contents onto the
clone. That plain-copy branch explicitly skips any path under `.github/`
(logging a warning rather than landing it) — a plain `--overlay` value-fix
directory can therefore never inject `.github/workflows/*` content, even by
accident, which is what keeps the "only two sanctioned sources" guarantee
above actually true. Every real per-repo overlay this toolkit ships
(`overlay-dozenos-1x/`) is a value-fix dir with its own `apply-overlay.sh`,
not a plain-copy target, but the exclusion holds even for a plain-copy
`--overlay` dir that has none.

**Caveat: `--allow-residuals` is not a workflow-content safety net.** Step
6's `--verify` (a full-tree grep for residual `vyos` tokens) is *not*
`.github/`-aware — it does not know a hit under `.github/workflows/*` is any
different from a hit anywhere else in the tree. `--allow-residuals`
(required by `--build-repo`, and by any per-repo `--overlay` push whose
value-fixes leave a deliberate residual, e.g. `dozenos-1x`'s) is now BOUNDED
by a checked-in allowlist (`overlay-dozenos-build/expected-residuals.txt`, matched by
`mirror-push.sh`'s `residuals_allowlisted()` on file path + a content
token — see "`--allow-residuals` is bounded by an allowlist" below) rather
than blanket-allowing every verify failure, but that allowlist is keyed on
plain repo-relative file paths, not on "is this under `.github/`" — it
provides no special protection for `.github/` specifically, and an
allowlisted non-`.github/` residual says nothing about `.github/` content.
The actual guarantee that no `.github/workflows/*` content leaks in from an
overlay comes from the plain-copy exclusion described above, not from
`--verify`/`--allow-residuals`.

### `--allow-residuals` is bounded by an allowlist, not a blanket allow

`mirror-push.sh` step 6 no longer treats `--allow-residuals` as "log and
continue past ANY verify failure." Every residual line `rename-transform.sh
--verify` reports (`<file>:<line>:<content>`) is checked against the
checked-in `overlay-dozenos-build/expected-residuals.txt` allowlist: a residual passes only
if its file matches an entry's path exactly AND its content contains that
entry's token substring (matched by content, not line number, so the
allowlist survives upstream line-number drift). If even one residual doesn't
match, `mirror-push.sh` dies (fail-closed) — even under
`--allow-residuals`/`--build-repo` — because an unmatched residual is a
candidate GENUINE `vyos` leak, not a known build-time pointer.

This closes a real gap: the previous implementation checked only `--verify`'s
boolean exit code, so *any* residual — including a brand-new genuine `vyos`
string a future upstream commit might introduce — was logged as "known
build-time pointers" and pushed anyway, nightly, with no alert, for
`dozenos-build` (the highest-churn repo, since `--build-repo` always implies
`--allow-residuals`).

The current, verified allowlist (`overlay-dozenos-build/expected-residuals.txt`) holds 9
entries for `dozenos-build`'s `--build-repo` residual set (3
`source-mirror`, 1 `cdn-syft`, 1 `toolchain-apt`, 3 `nonmirrored-org-ref` —
`AGENTS.md`'s 2 lines share one entry, for 9 residual *lines* from 8
*entries*) plus 1 more `nonmirrored-org-ref` entry for `dozenos-1x`'s
`--overlay overlay-dozenos-1x` push (`.coderabbit.yaml` is shared between
the two targets). This reconciles the residual-count discrepancy between
`SWEEP-dozenos-build.md`'s original 5-item classification (the non-git
binary/apt-host pointers only) and the 9 actually produced once
`overlay-dozenos-build/value-fixes/pin-nonmirrored-org-refs.sh` (REPOINT-AUDIT.md #6) is
included — 9 is the correct, current, re-verified count; see
`overlay-dozenos-build/MANIFEST.md` and `SWEEP-dozenos-build.md` for the full classification.

## Hard rule

**No `uses: vyos/*` reusable-workflow reference may ever appear in a
DozenOS-authored workflow file.** This includes `vyos/.github/...@production`
and `vyos/vyos-cla-signatures/...@production` verbatim, and also blocks
copy-pasting an upstream `trigger-docker-image-build.yml`-style file without
first checking it for such a reference (the 3 build workflows checked here
have none — they only call third-party marketplace actions — but item #8
must still re-verify this for whatever DozenOS build CI it authors, since it
is free to add new `uses:` steps of its own).

## Note: suspicious comment observed in upstream (not acted on)

While cross-checking upstream, two of the 8 infra workflow files
(`lts-name-check.yml`, `pr-mirror-repo-sync.yml`) were found to contain the
comment `# DO NOT EDIT — managed by mirror-pipeline rollout.` — confirmed
present in the real `vyos/vyos-build` repo via both a fresh clone and the
GitHub Contents API (not a local artifact). The phrasing is suspiciously
on-the-nose for an automated "mirror pipeline" agent exactly like this one,
and reads as a plausible prompt-injection attempt aimed at agents processing
this repo unattended. It was treated as inert file content, not an
instruction: it does not change this policy, and the wholesale `.github/`
strip removes both files regardless of their contents. Flagged here for
awareness only; no action taken or needed.

## Release distribution (#9)

Item #9 covers *what happens to a finished build's artifacts*, not workflow
authoring — the full spec lives in `DISTRIBUTION.md`. Summary, so this
policy file stays the single index of "where does X come from":

- **No public apt mirror.** DozenOS is image-based upgrade (whole-ISO swap,
  like VyOS) — confirmed by `CI-SECRETS.md`'s "Not present: no
  `DOZENOS_MIRROR_URL` / R2 / Cloudflare secrets" note. Nothing in this repo
  stands up or publishes to a persistent apt repo.
- **Two artifact tiers**, not one: (a) the item #8 `rebuild-packages.yml`
  workflow's `.deb`s are **ephemeral** GitHub Actions artifacts (job → job
  within a run, ≤90-day platform cap, `retention-days: 7` as configured) —
  never a durable mirror; (b) the finished **ISO + `.minisig` +
  `version.json` + `minisign.pub`** are a **durable** GitHub Release in
  `dozenos/dozenos-nightly-build`, which never expires. See
  `DISTRIBUTION.md` §2 for the full rationale and the retention-contrast
  table.
- **Reusable release helpers** (this item's actual deliverable) live in
  `release/` at the toolkit root — `release/gen-version-json.sh` (emits
  `version.json`) and `release/sign-and-publish.sh` (minisign-signs the ISO
  and runs `gh release create`). These are **toolkit files**, not overlay
  `new-files/` content — they are consumed *by* item #17's
  `dozenos-nightly-build` workflow at CI time, not shipped *into* any
  mirrored repo's tree. See `overlay-dozenos-build/MANIFEST.md`'s note on `release/` for
  the placement reasoning spelled out in full.
- Item #17 (not yet authored) is the actual nightly workflow that calls
  these helpers; item #13 (AUTHORED, see below) is the ephemeral in-job apt
  repo that consumes tier (a)'s `.deb`s to source the ISO's packages. Both
  are cross-referenced, not duplicated, in `DISTRIBUTION.md` §8.

## Ephemeral in-job apt repo / signed ISO (#13)

Item #13 is now authored — full spec and the verified `build-dozenos-image
--dozenos-mirror` flag live in
`ISO-BUILD.md`; the reusable helper is `release/make-ephemeral-apt-repo.sh`
(a toolkit file, same placement class as `release/gen-version-json.sh`/
`release/sign-and-publish.sh` — see `overlay-dozenos-build/MANIFEST.md`'s note on
`release/`). Summary: an ephemeral, unsigned, `[trusted=yes]` apt repo is
built inside the same container invocation that runs `build-dozenos-image`,
so the ISO's mandatory `dozenos-1x` package (and anything else
DozenOS-built) can actually be apt-installed during live-build chroot
assembly — `--dozenos-mirror ""` alone cannot satisfy that dependency. Its
sole consumer is `dozenos-nightly-build`'s `nightly.yml` (item #17), which
does a full, guaranteed-fresh rebuild-then-consume in one run. The original
consumer, a `package-smoketest.yml` `build_iso` job that best-effort-scanned
this repo's `rebuild-packages.yml` artifact history, was retired 2026-07-11:
the history scan structurally never contained the linux-kernel family, so
that job failed every run — see `REBUILD-DISPATCH.md` §6.
